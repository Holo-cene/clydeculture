# BE-09: Fix compute_dedupe_key session-timezone dependency

**Priority:** P2  
**Area:** Schema  
**Status:** Open  
**Depends on:** —

## Why this matters

`compute_dedupe_key()` is declared `IMMUTABLE` but uses
`TO_CHAR(DATE_TRUNC('hour', p_start_at), 'YYYY-MM-DD-HH24')`. For a `timestamptz`
input, `TO_CHAR` formats in the **session timezone** — not UTC. The same `timestamptz`
input produces different output depending on the `TimeZone` session variable. If a
migration runs in a developer's local environment during BST (`Europe/London`), it will
produce a different key for the same event than a production Edge Function running in
UTC. This would cause duplicate canonical records to be created at the unique constraint
and fail silently as apparent upsert conflicts.

## Prompt

You are building Clyde Culture. Read `docs/reference/SCHEMA_v5.sql` and
`docs/DEDUPLICATION.md` before proceeding.

The `compute_dedupe_key()` function in the schema has a timezone bug. Fix it with a
migration, and verify the `IMMUTABLE` declaration is correct after the fix.

**Step 1 — Create migration `supabase/migrations/20260601000007_fix_dedupe_key_timezone.sql`:**

Replace the function to force UTC normalisation before formatting:

```sql
create or replace function compute_dedupe_key(
  p_venue_id   uuid,
  p_start_at   timestamptz,
  p_title      text
)
returns text as $$
begin
  return encode(digest(
    coalesce(p_venue_id::text, 'no-venue')
    || '|'
    || to_char(
         date_trunc('hour', p_start_at at time zone 'UTC'),
         'YYYY-MM-DD-HH24'
       )
    || '|'
    || normalise_title(p_title),
    'sha256'
  ), 'hex');
end;
$$ language plpgsql immutable;
```

The `AT TIME ZONE 'UTC'` cast normalises the input to UTC before truncation, making
the output independent of session timezone. The `IMMUTABLE` declaration is now
genuinely correct: given the same inputs, the function always returns the same output
regardless of server configuration.

**Step 2 — Check for existing rows:**

Add a comment in the migration noting: "If any `events` rows already exist with keys
computed under the old function (session-tz-aware), they must be recomputed. Run
`UPDATE events SET dedupe_key = compute_dedupe_key(venue_id, start_at, normalised_title)`
after applying this migration to production if data already exists."

**Step 3 — Update `docs/DEDUPLICATION.md`:**

In the "The dedupe_key" section, update the code block to show `AT TIME ZONE 'UTC'`
in the `TO_CHAR` call, with a note: "Start time is normalised to UTC before bucketing
so the hash is independent of session timezone."
