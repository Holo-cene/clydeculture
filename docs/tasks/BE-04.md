# BE-04: Handle rescheduled events and dedupe_key changes

**Priority:** P1  
**Area:** Ingestion  
**Status:** Open  
**Depends on:** BE-03

## Why this matters

`dedupe_key` is SHA-256 of `venue_id | start_bucket | normalised_title`. When a
Ticketmaster event is rescheduled to a new date, the `start_at` changes, the bucket
changes, and therefore the key changes. On the next ingestion run the normaliser
computes a new key, finds no matching canonical event, and inserts a duplicate at
`visibility = 'draft'`. The original canonical event remains at `visibility = 'published'`
with the wrong date until `archive_past_events()` eventually clears it. This is a
correctness bug that affects real-world events regularly.

## Prompt

You are building Clyde Culture. Read `docs/DEDUPLICATION.md`, `docs/INGESTION.md`,
`docs/DATA_MODEL.md`, and `docs/reference/SCHEMA_v5.sql` before proceeding.

The cross-source deduplication uses a `dedupe_key` stored on both `events` and
implicitly derived from `external_events` during normalisation. When source data
changes in a way that shifts the key (date reschedule, venue change), the normaliser
currently creates a new canonical event rather than updating the existing one.

**Your task** is to add a mechanism for detecting and handling key-change events.

**Step 1 — Add `previous_dedupe_key` to `external_events`:**

Create migration `supabase/migrations/20260601000003_previous_dedupe_key.sql`:

```sql
alter table external_events
  add column if not exists previous_dedupe_key text;
```

This column stores the last key the normaliser computed for this external record. On the
next run, if the newly computed key differs from `previous_dedupe_key`, a key-change
has occurred.

**Step 2 — Add a Postgres function `handle_dedupe_key_change()`:**

In the same migration:

```sql
create or replace function handle_dedupe_key_change(
  p_external_event_id uuid,
  p_new_dedupe_key     text
) returns void as $$
-- Find the existing canonical event via external_events.event_id
-- Update events.dedupe_key to p_new_dedupe_key
-- Update events.start_at and events.end_at from the external_events row
-- Update the denormalised slug on events (use new start_at in slug convention)
-- Set availability = 'rescheduled' if start_at changed, or update venue fields if venue changed
-- Write previous_dedupe_key = old dedupe_key before update
$$ language plpgsql;
```

Implement fully. The function should NOT create a new `events` row; it updates the
existing one in place. The old `dedupe_key` value is overwritten on `events` (the
unique index will need to be temporarily deferred or the update done atomically).

**Step 3 — Document the key-change lifecycle in `docs/DEDUPLICATION.md`:**

Add a section "Dedupe key changes (rescheduled and relocated events)" after the
"Fuzzy-match candidates" section. Explain:
- When a key change is detected (new key ≠ `previous_dedupe_key`)
- That `handle_dedupe_key_change()` updates the existing canonical event in place
- That `availability` is set to `'rescheduled'` automatically when `start_at` changes
- That no duplicate canonical event is created

Also note the edge case: if the new key collides with a *different* existing canonical
event (two events effectively swapping identity), treat this as a merge candidate and
write to `event_merge_candidates` rather than attempting an in-place update.
