# DB-05: Harden event_submissions RLS and add venue_claims Phase 2 policies

**Priority:** P2  
**Area:** Schema, Security  
**Status:** Open  
**Depends on:** DB-13 (auth model ADR — for venue_claims authenticated policy)

## Why this matters

Two RLS gaps exist in the community tables:

1. `event_submissions` has a `FOR INSERT WITH CHECK (true)` policy — no field-level
   sanity check, no structural validation at the DB layer. A bot can insert a submission
   with `start_at = '1900-01-01'`, a 10,000-character title, or a blank title and it
   passes silently into the moderation queue.

2. `venue_claims` has RLS enabled but no policies defined. Default-deny means any INSERT
   from an authenticated user — the entire Phase 2 community claim feature — fails with a
   policy error before a single line of application code is written.

The `event_submissions` hardening can be done now. The `venue_claims` INSERT policy
depends on the auth model defined in DB-13 (the Supabase Auth ADR). If DB-13 is not yet
resolved, implement only the `event_submissions` constraint changes and leave a TODO
comment for the `venue_claims` policy.

### Part 1 — event_submissions RLS hardening (do this now)

**Create migration `supabase/migrations/20260601000023_rls_community_tables.sql`:**

Replace the current `event_submissions` INSERT policy with one that includes structural
validation:

```sql
-- Drop the existing unconstrained policy
drop policy if exists "Public insert submissions" on event_submissions;

-- Replacement: structural sanity checks at the DB boundary.
-- Application-layer CAPTCHA and rate limiting must still be implemented separately.
create policy "Public insert submissions" on event_submissions
  for insert with check (
    -- Title must be present and reasonable length
    length(trim(title)) >= 3
    and length(title) <= 300
    -- Start time cannot be in the distant past (likely spam/test data)
    and start_at >= now() - interval '1 day'
    -- Start time cannot be unreasonably far in the future
    and start_at <= now() + interval '2 years'
    -- submitter_email, if provided, must look like an email
    and (
      submitter_email is null
      or submitter_email ~* '^[^@\s]+@[^@\s]+\.[^@\s]+$'
    )
  );
```

Add a comment block above the policy:

```sql
-- event_submissions: public insert only.
-- These WITH CHECK conditions are a first line of defence against structurally
-- invalid submissions. They do not replace CAPTCHA or rate limiting, which must
-- be implemented in the Edge Function or form handler.
-- No SELECT policy: submitters cannot read back their submission.
-- Operators access via service role.
```

### Part 2 — venue_claims INSERT policy (add after DB-13 is resolved)

Once `docs/decisions/0003-auth-model.md` (DB-13) exists and names the auth pattern,
add the following to the same migration file (or a new migration if DB-13 is separate):

```sql
-- venue_claims: authenticated users can submit a claim for a claimable venue.
-- No SELECT policy for claimants — operators review via service role.
create policy "Authenticated users can claim venues" on venue_claims
  for insert with check (
    auth.role() = 'authenticated'
    -- Can only claim venues marked as claimable
    and exists (
      select 1 from venues
      where venues.id = venue_id
        and venues.claimable = true
        and venues.status in ('active', 'temporary')
    )
  );
```
