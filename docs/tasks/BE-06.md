# BE-06: Specify and implement the fuzzy-match similarity threshold

**Priority:** P2  
**Area:** Deduplication  
**Status:** Open  
**Depends on:** BE-03

## Why this matters

`docs/DEDUPLICATION.md` describes a `pg_trgm` trigram similarity pass that writes
near-duplicate pairs to `event_merge_candidates` — but the similarity threshold value
appears nowhere in the docs or schema. Any engineer or agent implementing the
deduplication pipeline will choose their own value. The worked example in the doc
("Sub Club: Optimo" / "Optimo at Sub Club") implies the threshold must be below ≈0.48
to catch that pair, but a threshold of 0.30 would create massive false-positive volume.
An unspecified threshold is an implementation trap.

---

## Prompt

You are building Clyde Culture. Read `docs/DEDUPLICATION.md`, `docs/DATA_MODEL.md`,
and `docs/reference/SCHEMA_v5.sql` before proceeding.

**Your task** is to define the fuzzy-match threshold and surface it as a
per-source configurable value.

**Step 1 — Add `fuzzy_threshold` to the `sources` table:**

Create migration `supabase/migrations/20260601000005_fuzzy_threshold.sql`:

```sql
alter table sources
  add column if not exists fuzzy_threshold numeric(4,3) not null default 0.35
  check (fuzzy_threshold between 0.1 and 1.0);
```

Rationale for 0.35: the "Sub Club: Optimo" / "Optimo at Sub Club" trigram similarity
is ≈0.48; a threshold of 0.35 catches that pair with margin. Setting it higher (0.5+)
would miss word-order variations; lower (0.2) produces too many false positives at
venue/hour level. Per-source configuration allows Tier 3 HTML scrapers (which have
less consistent titling) to use a lower threshold without affecting Tier 1 API pairs.

**Step 2 — Add a Postgres function `find_fuzzy_merge_candidates()`:**

In the same migration:

```sql
create or replace function find_fuzzy_merge_candidates(p_event_id uuid)
returns void as $$
-- Given a newly created or updated canonical event, find all events that:
-- 1. Share the same venue_id
-- 2. Have start_at within the same hourly bucket
-- 3. Have dedupe_key != this event's dedupe_key (not already exact-matched)
-- 4. Have normalised_title similarity >= the source's fuzzy_threshold
-- For each match above threshold, upsert a row into event_merge_candidates
-- using the LEAST/GREATEST pattern on event IDs to prevent duplicate pairs.
$$ language plpgsql;
```

Implement fully. Use `similarity(e.normalised_title, target.normalised_title)` from
`pg_trgm`. The source's threshold is obtained by joining `external_events` back to
`sources` via the canonical event's `primary_source_id`.

**Step 3 — Update `docs/DEDUPLICATION.md`:**

In the "Fuzzy-match candidates" section, add:
- The default threshold value (0.35) and the rationale
- A note that threshold is configurable per source via `sources.fuzzy_threshold`
- An explicit statement that the function runs once per newly created/updated event
- The worked examples with computed similarity scores to validate the threshold choice
