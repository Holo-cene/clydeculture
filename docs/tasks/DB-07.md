# DB-07: Add publish_mappings delete guard and events hard-delete prevention

**Priority:** P2  
**Area:** Schema, Publishing  
**Status:** Open  
**Depends on:** —

## Why this matters

`publish_mappings.entity_id` is a polymorphic UUID with no foreign key. If a canonical
`events` row is hard-deleted (e.g., during a merge cleanup), its `publish_mappings` row
persists. The Webflow sync job uses `publish_mappings` to detect deletions: it compares
the mapping to the Postgres entity to decide whether to remove the Webflow item. With
the Postgres row gone, the sync job either errors or silently skips the deletion.
The Webflow CMS retains a stale published item indefinitely.

The correct lifecycle is: set `visibility = 'archived'` → sync job detects the
transition → removes the Webflow item → mapping is cleaned up. Hard-deleting an event
bypasses this entirely. A DB trigger that blocks hard-deletes on events with active
publish mappings enforces the safe lifecycle at the database level.

**Create migration `supabase/migrations/20260601000025_publish_mappings_guard.sql`:**

```sql
-- Prevent hard-deleting a canonical event that has an active publish mapping.
-- The correct lifecycle is: set visibility = 'archived', let the sync job
-- remove the Webflow item and clean up the mapping, then optionally delete.
--
-- This trigger fires before DELETE on events. If a publish_mappings row exists
-- for the event (entity_type = 'event', entity_id = events.id), it raises an
-- exception. The operator must either:
--   (a) archive the event and wait for the sync job to remove the mapping, or
--   (b) manually delete the publish_mappings row first if the Webflow item has
--       already been removed independently.
create or replace function prevent_published_event_delete()
returns trigger as $$
begin
  if exists (
    select 1 from publish_mappings
    where entity_type = 'event'
      and entity_id = old.id
  ) then
    raise exception
      'Cannot hard-delete event % (slug: %) — a publish_mappings row exists. '
      'Set visibility = ''archived'' and run the Webflow sync job first, or '
      'delete the publish_mappings row manually if the Webflow item is already gone.',
      old.id, old.slug;
  end if;
  return old;
end;
$$ language plpgsql;

create trigger trg_prevent_published_event_delete
  before delete on events
  for each row execute function prevent_published_event_delete();
```

**Update `docs/DATA_MODEL.md`:**

Add a subsection "Event deletion lifecycle" to the Publishing section:

> **Event deletion lifecycle**
>
> `events` rows are never hard-deleted while an active `publish_mappings` row exists.
> The database enforces this via `trg_prevent_published_event_delete`. The correct
> lifecycle for removing an event from the frontend is:
>
> 1. Set `visibility = 'archived'` on the event.
> 2. Run the Webflow sync job — it detects the `archived` transition and removes the
>    Webflow CMS item.
> 3. The sync job deletes the `publish_mappings` row after confirming deletion.
> 4. The `events` row may now be hard-deleted if required (e.g., confirmed spam), or
>    left in place as an archive record.
>
> For events that were never published to Webflow (no mapping row exists), hard-delete
> is permitted at any time.

**Update `docs/OPERATIONS.md`** Moderation Queue section — add a note after the merge

> When merging duplicate events: set the losing event to `visibility = 'hidden'` rather
> than deleting it. If the losing event has a `publish_mappings` row, the sync job will
> remove the Webflow item. Hard-deletion while a mapping exists is blocked by a database
> trigger.
