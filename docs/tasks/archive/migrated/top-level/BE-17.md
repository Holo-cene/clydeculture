# BE-17: Extend ingest_alerts for festival detection failures

**Priority:** P2  
**Area:** Schema  
**Status:** Open  
**Depends on:** BE-16

## Why this matters

`docs/FESTIVALS.md` says: "events that fail the date-window check are not discarded —
they are stored as ordinary events with `festival_id` null, and an alert is logged so
the condition can be investigated manually." However, the `ingest_alerts` table's
`alert_type` CHECK constraint only allows `('count_drop', 'parse_failure', 'timeout',
'manual')`. There is no `festival_window_mismatch` type. Any code that attempts to
insert this alert will violate the constraint and either throw an error or be silently
swallowed — either way, the alert the docs promise is never written.

## Prompt

You are building Clyde Culture. Read `docs/FESTIVALS.md`, `docs/reference/SCHEMA_v5.sql`,
and `docs/DATA_MODEL.md` before proceeding.

**Your task** is to extend the `ingest_alerts` constraint to include festival detection
alert types, and update all related documentation.

**Step 1 — Create migration `supabase/migrations/20260601000012_ingest_alerts_festival.sql`:**

Extend the `alert_type` CHECK constraint on `ingest_alerts`:

```sql
-- Add festival detection alert types to ingest_alerts.
-- Current allowed values: count_drop, parse_failure, timeout, manual
-- New values: festival_window_mismatch (event matched festival rules but outside date window)

alter table ingest_alerts
  drop constraint if exists ingest_alerts_alert_type_check;

alter table ingest_alerts
  add constraint ingest_alerts_alert_type_check
  check (alert_type in (
    'count_drop',
    'parse_failure',
    'timeout',
    'manual',
    'festival_window_mismatch'   -- event matched a festival rule but fell outside start/end window
  ));
```

**Step 2 — Update `docs/FESTIVALS.md`:**

In the "Date-Window Validation" section, replace "An alert is logged" with:

> An `ingest_alerts` row is written with `alert_type = 'festival_window_mismatch'`,
> recording the `source_id`, a null `run_id` (window checks are not tied to a single run),
> and a `message` of the form:
> `"Festival match 'celtic-connections-2027' for event <slug> failed window check: event 2025-10-15 outside 2026-01-16–2026-02-02"`.
> The alert is resolved by either adding the event to a manual override
> (`festival_event_overrides`) or confirming it is correctly untagged.

**Step 3 — Update `docs/DATA_MODEL.md`:**

In the `ingest_alerts` section, update the `alert_type` description to include
`festival_window_mismatch` alongside the existing types, with a brief definition.
