# DB-02: Add PostGIS extension and venue geometry column for map queries

**Priority:** P1  
**Area:** Schema  
**Status:** Open  
**Depends on:** —

## Why this matters

`SPEC.md` §3 includes "Venue pages and map view" in Phase 1. The current schema has a
`(lat, lng)` B-tree index on `venues`, which supports equality and range scans on each
axis independently. It cannot support proximity queries ("venues within 500 m of this
point", "venues in this bounding box") — those require PostGIS. Without PostGIS, either
the venue map feature cannot be implemented correctly, or the application must pull every
venue row and filter in application memory. PostGIS is available on all Supabase plans
and is a single-extension addition.

---

## Prompt

You are building Clyde Culture. Read `docs/reference/SCHEMA_v5.sql`,
`docs/DATA_MODEL.md` (venues section), and `docs/reference/SPEC.md` §3 before
proceeding.

The venue map feature requires spatial query support. Add PostGIS, a generated geometry
column, and a GiST index to support radius and bounding-box queries on the `venues`
table.

**Create migration `supabase/migrations/20260601000021_venue_postgis.sql`:**

```sql
-- Enable PostGIS. Available on all Supabase plans.
create extension if not exists postgis;

-- Add a generated geometry column.
-- Computed from lng (x) and lat (y) in WGS-84 (SRID 4326).
-- Only populated when both lat and lng are non-null.
alter table venues
  add column if not exists geom geometry(Point, 4326)
    generated always as (
      case
        when lat is not null and lng is not null
        then ST_SetSRID(ST_MakePoint(lng::float8, lat::float8), 4326)
        else null
      end
    ) stored;

-- GiST index for proximity and bounding-box queries.
create index idx_venues_geom on venues using gist (geom)
  where geom is not null;

-- Drop the redundant B-tree lat/lng index; the GiST index supersedes it
-- for all realistic query patterns against this table at Phase 1 scale.
drop index if exists idx_venues_geo;
```

> Note: the existing `idx_venues_geo` index is a B-tree on `(lat, lng)`. It is
> redundant once the GiST index exists for spatial queries, and it adds write overhead
> without benefit. Drop it.

**Update `docs/DATA_MODEL.md`:**

In the `venues` table description, add `geom` to the column table:

| Column | Type | Notes |
|---|---|---|
| geom | geometry(Point, 4326) | Generated from lat/lng; null when either is missing. Used for map proximity queries via `ST_DWithin`. |

Add a note below the column table:

> Spatial queries use PostGIS. Example: venues within 1 km of a point:
> ```sql
> select * from venues
> where ST_DWithin(
>   geom,
>   ST_SetSRID(ST_MakePoint(-4.2518, 55.8642), 4326)::geography,
>   1000  -- metres
> )
> and status in ('active', 'temporary');
> ```
> The cast to `::geography` makes the distance unit metres rather than degrees.

**Update `docs/reference/SCHEMA_v5.sql`:**

Add `create extension if not exists postgis;` to the Extensions section alongside
`pgcrypto` and `pg_trgm`.

**Do not modify any other tables or add any other extensions.**
