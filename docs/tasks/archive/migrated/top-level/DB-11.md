# DB-11: Multi-room venue strategy for SWG3 — spike and alias seed

**Priority:** P2  
**Area:** Schema, Connectors  
**Status:** Open  
**Depends on:** —

## Why this matters

SWG3 is explicitly listed as a Phase 1 venue (HTML scraper) in `SPEC.md §7`, and the R2
review prompt specifically names it as a multi-room venue risk. SWG3 runs at least four
simultaneously active spaces: Galvanize, The Studio, The Yard, and The Arts Club. Two
risks exist:

1. **False merge:** Two simultaneous events at SWG3 with near-identical titles (unlikely
   but not impossible) would produce the same dedupe_key and the second insert would
   silently overwrite the first canonical record.

2. **Orphan venue proliferation:** If Ticketmaster or Skiddle lists "SWG3 Galvanize" and
   "SWG3 Studio" as separate venue strings (which they sometimes do), `resolve_venue`
   returns null for both, `auto_create_venue` creates two bare stub rows, and two
   separate canonical events are created for what is really one event at one venue. The
   merge candidate queue fills with SWG3 noise on every ingestion run.

The solution is to seed `venue_aliases` with all known SWG3 name variants pointing to
one canonical `venues` row, so `resolve_venue` consistently returns the same UUID
regardless of which API's venue string comes through the pipeline.

### Step 1 — Canonical venue seed for SWG3

Create `supabase/migrations/20260601000027_venue_seed_swg3.sql`:

```sql
-- Canonical SWG3 venue row.
-- SWG3 is a multi-space venue in Glasgow's south side. Its sub-spaces
-- (Galvanize, The Studio, The Yard, The Arts Club) all map to this single
-- venue row. The dedupe_key for simultaneous events in different spaces is
-- therefore differentiated by title only.
insert into venues (
  name, slug, status, city, auto_created, needs_review,
  website, claimable
) values (
  'SWG3',
  'swg3',
  'active',
  'Glasgow',
  false,
  false,
  'https://swg3.tv',
  true
)
on conflict (slug) do nothing;

-- All known venue name variants from APIs and scrapers.
-- Add new variants here as they are discovered during connector development.
-- Each normalised_alias must be unique; use lower(trim(variant)) as the key.
insert into venue_aliases (venue_id, alias, normalised_alias)
select
  v.id,
  alias_data.alias,
  alias_data.normalised_alias
from venues v,
(values
  ('SWG3',                   'swg3'),
  ('SWG3 Glasgow',           'swg3 glasgow'),
  ('SWG3 - Galvanize',       'swg3  galvanize'),
  ('SWG3 Galvanize',         'swg3 galvanize'),
  ('Galvanize at SWG3',      'galvanize at swg3'),
  ('SWG3 - The Studio',      'swg3  the studio'),
  ('SWG3 The Studio',        'swg3 the studio'),
  ('SWG3 Studio',            'swg3 studio'),
  ('SWG3 - The Yard',        'swg3  the yard'),
  ('SWG3 The Yard',          'swg3 the yard'),
  ('SWG3 - The Arts Club',   'swg3  the arts club'),
  ('SWG3 The Arts Club',     'swg3 the arts club'),
  ('SWG3 Arts Club',         'swg3 arts club')
) as alias_data(alias, normalised_alias)
where v.slug = 'swg3'
on conflict (normalised_alias) do nothing;
```

> Note: the `normalised_alias` values above use `lower(trim(alias))`. Verify each one
> matches the output of `lower(trim(original_string))` exactly — there should be no
> double-spaces after the dash/hyphen variants are stripped of punctuation by
> `normalise_title`. Adjust if needed to match what `resolve_venue` would compute.

Actually — `resolve_venue` compares `lower(trim(p_venue_name))` against
`normalised_alias`. It does NOT call `normalise_title()` (which strips punctuation).
So "SWG3 - Galvanize" normalises to `'swg3 - galvanize'` (not `'swg3  galvanize'`).
The aliases must be exact `lower(trim())` of the source string. Correct the values to
match what the APIs actually return — verify against live API responses during connector
development and add aliases to this seed accordingly.

### Step 2 — Add a venue-aliases maintenance note to OPERATIONS.md

Add a subsection "Venue alias maintenance" to `docs/OPERATIONS.md` under Moderation Queue:

> **Venue alias maintenance.** When an ingestion run creates a bare `venues` row via
> `auto_create_venue()`, check whether the new venue is a sub-space or variant name of
> an existing venue. If so:
>
> 1. INSERT a row into `venue_aliases` linking the new name variant to the canonical
>    venue row.
> 2. Delete the auto-created bare venue row (it has no events linked yet if caught
>    early, or update `external_events.venue_id_guess` and rerun normalisation if events
>    already exist).
> 3. Add the alias to the relevant seed migration file so it is not lost on next reset.
>
> For major multi-space venues (SWG3, SEC/Hydro, Barrowland), seed all known API name
> variants upfront. See `supabase/migrations/20260601000027_venue_seed_swg3.sql` as
> an example.

### Step 3 — Update docs/DEDUPLICATION.md

In the "Venue" subsection of Stage 2, add a paragraph:

> **Multi-space venues.** Venues that operate multiple simultaneous spaces under one
> brand (SWG3, SEC/Hydro) present a specific dedup risk: two events running at the same
> time in different rooms produce the same venue_id component in the dedupe_key and are
> differentiated only by title normalisation. In practice, events in different rooms
> have different titles. However, the more common failure mode is that APIs return
> sub-space names ("SWG3 Galvanize", "SWG3 Studio") as separate venue strings, causing
> `resolve_venue` to miss them and `auto_create_venue` to create orphan stubs. This is
> mitigated by pre-seeding all known alias variants for multi-space venues in
> `venue_aliases` before the first ingestion run. See the seed migration for SWG3 as
> the reference pattern.
