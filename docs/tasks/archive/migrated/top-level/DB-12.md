# DB-12: Define Phase 2 auth model (Supabase Auth, roles, RLS patterns)

**Priority:** P2 (Spike)  
**Area:** Schema, Security  
**Status:** Open  
**Depends on:** —

## Why this matters

The schema has RLS enabled on all 20 tables with 9 policies, but every write policy is
either "anon insert" (`event_submissions`) or "service role only" (everything else).
There are no `auth.uid()`-scoped policies anywhere. Phase 2 requires:

- Authenticated operators for moderation (reviewing submissions, merge candidates, venue claims)
- Authenticated venue operators for claim approval and profile editing
- Possibly authenticated submitters who want to track their own submissions

None of these can be built without defining: which Supabase Auth providers to use,
how admin/operator roles are assigned, and what each table's authenticated-user RLS
policy looks like. Without an ADR, Phase 2 features will be designed independently
by different contributors and the RLS layer will be inconsistent.

This is a spike: the output is an ADR that unblocks DB-05 (venue_claims policy) and
all Phase 2 community feature development.

### Step 1 — Write the ADR

Create `docs/decisions/0003-auth-model.md` using the template at
`docs/decisions/0000-adr-template.md`. The ADR must answer:

1. **Which Supabase Auth provider(s) are used?** Options: email + magic link (recommended
   for a low-volume admin use case), GitHub OAuth, Google OAuth, or a custom JWT issuer.
   For Clyde Culture's use case (small team of operators + venue operators), email
   magic link is the simplest and requires no OAuth app setup. Justify the choice.

2. **How are admin/operator roles assigned?** Options:
   - Custom JWT claims set via a Supabase Auth hook (`custom_claims` JSON in the JWT)
   - A separate `user_roles` table joined in RLS policies (`auth.uid()` → role lookup)
   - Supabase built-in roles (limited options: `authenticated` vs `service_role`)
   
   Recommend one approach and explain the operational process for granting operator
   access (e.g., "run a SQL function that sets custom claims for a given user").

3. **What does each Phase 2 table's RLS policy look like?**
   Write out the policy intent (not full SQL yet) for:
   - `event_submissions` (SELECT for operators; UPDATE for operators; SELECT own row for submitters)
   - `venue_claims` (INSERT for authenticated users; SELECT/UPDATE for operators)
   - `moderation_log` (SELECT for operators; INSERT for operators)
   - `venues` (UPDATE for venue operators who have an approved `venue_claim`)

4. **What is the process for onboarding a new operator or venue operator?** Document it
   in the ADR in enough detail that it can be executed without reading Supabase docs.

### Step 2 — Implement the RLS migration

After the ADR is written and the design confirmed, create
`supabase/migrations/20260601000028_rls_auth_model.sql` implementing the policies
described in the ADR for the Phase 2 tables.

The migration should use the pattern chosen in the ADR for role detection. Example
(if using custom JWT claims):

```sql
-- Helper function: true if the current user has the 'operator' role claim
create or replace function is_operator()
returns boolean as $$
  select coalesce(
    (auth.jwt() -> 'app_metadata' ->> 'role') = 'operator',
    false
  );
$$ language sql stable security definer;

-- Operators can SELECT all event_submissions
create policy "Operators read submissions" on event_submissions
  for select using (is_operator());

-- Operators can UPDATE (approve/reject) submissions
create policy "Operators update submissions" on event_submissions
  for update using (is_operator());

-- ... and so on for each table listed in the ADR
```
