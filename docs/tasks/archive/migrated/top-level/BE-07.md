# BE-07: Add an incremental-sync cursor to the Connector interface

**Priority:** P2  
**Area:** Connectors  
**Status:** Open  
**Depends on:** BE-01

## Why this matters

`Connector.run()` takes no parameters. The orchestrator always does a full fetch —
no mechanism exists to pass a "since last run" cursor to connectors that support it.
Ticketmaster, Skiddle, and Eventbrite all support date-range or modified-since filters.
Without a cursor, the daily run fetches the entire Glasgow event catalogue every time,
burning API quota unnecessarily and making runs slower than they need to be.

---

## Prompt

You are building Clyde Culture. Read `docs/CONNECTOR_GUIDE.md` and `docs/INGESTION.md`
before proceeding. The connector interface definition is described in
`docs/CONNECTOR_GUIDE.md` — the actual TypeScript file will live at
`packages/connectors/src/connector.ts` once the monorepo is initialised (BE-01).

**Your task** is to update the connector interface and documentation to support an
optional incremental-sync cursor, without breaking connectors that don't use it.

**Step 1 — Update the interface definition in `docs/CONNECTOR_GUIDE.md`:**

Change the `Connector` interface in section 3 to:

```ts
export interface RunContext {
  since?: Date;  // when set, fetch only records modified/created after this date
}

export interface Connector {
  readonly slug: string;
  readonly type: SourceType;
  run(ctx?: RunContext): Promise<IngestResult>;
}
```

The `since` parameter is optional. Connectors that do not support incremental sync
ignore `ctx` entirely. Connectors that do support it use `ctx?.since` to add a
`modified_since` or `start_date_time` filter to their upstream API call.

**Step 2 — Update the orchestrator description in `docs/INGESTION.md`:**

In step 3 of the "Scheduled-job model" section, change:

> 3. Calls the connector's `run()` method and collects the `IngestResult`.

to:

> 3. Calls the connector's `run({ since: sources.last_success_at })` method. If
>    `last_success_at` is null (first run), passes `run()` with no context, triggering a
>    full historical fetch. If `last_success_at` is set, passes it as `since` so
>    connectors that support incremental sync fetch only new and changed records.

**Step 3 — Update the worked example (RSS connector) in `docs/CONNECTOR_GUIDE.md`:**

Show the correct `run(ctx?: RunContext)` signature in the RSS example. The RSS
connector does not use `ctx` (feeds do not support since-filters), so the implementation
body is unchanged — only the signature needs updating to show the accepted but ignored
parameter:

```ts
async run(_ctx?: RunContext): Promise<IngestResult> {
  // RSS feeds do not support incremental sync; full fetch every run.
  ...
}
```

**Step 4 — Add a note in section 1 (Choose a source type)** indicating which source
types support the `since` parameter: `api` connectors typically do; `rss`, `ical`, and
`html` connectors typically do not, and should accept the parameter but ignore it.
