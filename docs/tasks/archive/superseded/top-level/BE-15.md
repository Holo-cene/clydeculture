# BE-15: Specify when the fuzzy dedup pass runs

**Priority:** P2  
**Area:** Deduplication  
**Status:** Open  
**Depends on:** BE-06

## Why this matters

`docs/DEDUPLICATION.md` describes the fuzzy-match secondary pass as running after the
dedupe_key step — but never says *when* in the ingestion pipeline it executes. Does it
run after each connector's results are committed, once after the full daily sweep, or as
a separate scheduled job? The answer matters: if it runs per-connector, a Ticketmaster
event and a matching SWG3 scraper event will never compare against each other within
the same run (each runs on its own schedule). If it runs once per day after all
connectors, the candidates window could be many hours old by the time it fires.

## Prompt

You are building Clyde Culture. Read `docs/INGESTION.md`, `docs/DEDUPLICATION.md`,
and `docs/decisions/0003-orchestration-scheduling.md` (created in BE-05) before
proceeding.

**Your task** is to specify exactly when the fuzzy dedup pass runs and add it to the
orchestrator description.

**Decision to make and document:**

The correct approach is to run the fuzzy pass as a **separate scheduled job** that runs
once per day, after all connector runs have completed. Reasons:
- API and HTML scraper connectors run on staggered schedules (BE-05); no single
  connector run has all sources' events available for comparison.
- Running per-event on insert would require comparing against all events at the same
  venue/bucket on every upsert — expensive and fragile under concurrent writes.
- A once-daily pass over all events created or updated in the last 24 hours is bounded,
  auditable, and easy to re-run.

**Step 1 — Update `docs/INGESTION.md`:**

Add a new section "Fuzzy deduplication pass" after "Break detection":

> The fuzzy deduplication pass is a separate scheduled job that runs daily, after all
> connector runs have completed (suggested time: 06:00 UTC, after the last connector
> completes). It calls `find_fuzzy_merge_candidates()` (see BE-06) for every `events`
> row whose `updated_at` is within the last 48 hours — a 48-hour window ensures that
> events normalised by late-running connectors are always included. Candidates above
> the source's `fuzzy_threshold` are written to `event_merge_candidates` with
> `status = 'pending'`.
>
> The pass does not merge automatically. Merge candidates surface in the moderation
> queue for human review. The pass is idempotent: re-running it does not create
> duplicate `event_merge_candidates` rows because the unique constraint on
> `(LEAST(event_a_id, event_b_id), GREATEST(event_a_id, event_b_id))` prevents
> duplicate pairs.

**Step 2 — Add the cron entry to the scheduling migration (from BE-05):**

Amend `supabase/migrations/20260601000004_per_connector_cron.sql` (or create a new
migration) to add a fuzzy dedup cron entry:

```sql
select cron.schedule(
  'fuzzy-dedup-pass',
  '0 6 * * *',   -- 06:00 UTC daily, after all connectors have run
  $$ select net.http_post(
    url  := current_setting('app.edge_function_url') || '/fuzzy-dedup',
    body := '{}'::jsonb
  ) $$
);
```

**Step 3 — Update `docs/DEDUPLICATION.md`:**

In the "Fuzzy-match candidates" section, add an opening sentence:
"The fuzzy pass runs as a separate daily job at 06:00 UTC. It checks all events
updated within the last 48 hours against other events at the same venue and time
bucket."
