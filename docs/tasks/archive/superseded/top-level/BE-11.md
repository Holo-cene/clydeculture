# BE-11: Add venue-rename propagation for denormalised fields

**Priority:** P2  
**Area:** Schema  
**Status:** Open  
**Depends on:** —

## Why this matters

The `events` table carries 13 denormalised fields for Webflow, including
`venue_name_display` and `venue_slug_display`. These exist because Webflow CMS cannot
perform joins. `DATA_MODEL.md` acknowledges: "when a venue renames, all events at that
venue need `venue_name_display` updated." No trigger or utility function exists to do
this. A maintainer who updates a venue name directly in Supabase Studio will silently
produce inconsistent state — all existing events will show the old venue name — with
no error and no alert. `validate_event_consistency()` checks consistency but does not
fix it.

## Prompt

You are building Clyde Culture. Read `docs/DATA_MODEL.md`, `docs/PUBLISHING.md`, and
`docs/reference/SCHEMA_v5.sql` before proceeding.

**Your task** is to add a Postgres function (and optionally a trigger) that propagates
venue field changes to all denormalised display fields on linked events.

**Step 1 — Create migration `supabase/migrations/20260601000008_venue_rename_propagation.sql`:**

Add a function `resync_venue_display_fields(p_venue_id uuid)`:

```sql
create or replace function resync_venue_display_fields(p_venue_id uuid)
returns integer as $$
-- Updates venue_name_display, venue_slug_display, and location_display on all
-- events rows where venue_id = p_venue_id.
-- Returns count of events updated.
declare
  v_venue record;
  updated_count integer;
begin
  select name, slug, address, city, postcode
  into v_venue
  from venues where id = p_venue_id;

  if v_venue is null then
    return 0;
  end if;

  update events
  set
    venue_name_display = v_venue.name,
    venue_slug_display = v_venue.slug,
    location_display   = coalesce(
                           trim(v_venue.address || ', ' || v_venue.city),
                           v_venue.city,
                           'TBA'
                         )
  where venue_id = p_venue_id;

  get diagnostics updated_count = row_count;
  return updated_count;
end;
$$ language plpgsql;
```

Also add an `AFTER UPDATE ON venues` trigger that calls this function automatically
when `name`, `slug`, `address`, `city`, or `postcode` changes:

```sql
create or replace function trigger_resync_venue_display()
returns trigger as $$
begin
  if (new.name    is distinct from old.name    or
      new.slug    is distinct from old.slug    or
      new.address is distinct from old.address or
      new.city    is distinct from old.city    or
      new.postcode is distinct from old.postcode) then
    perform resync_venue_display_fields(new.id);
  end if;
  return new;
end;
$$ language plpgsql;

create trigger resync_venue_display_after_update
  after update on venues
  for each row execute function trigger_resync_venue_display();
```

**Step 2 — Update `docs/DATA_MODEL.md`:**

In the "Denormalised Fields on events" section, add: "When a venue record is updated,
the `resync_venue_display_after_update` trigger automatically propagates name, slug,
and location changes to all linked events. The function can also be called manually:
`SELECT resync_venue_display_fields('<venue_uuid>')` returns the number of events
updated."

Add `resync_venue_display_fields()` to the Helper Functions table at the bottom.
