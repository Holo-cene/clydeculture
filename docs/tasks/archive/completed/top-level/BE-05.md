# BE-05: Document the fan-out approach for long connector sweeps

**Priority:** P2  
**Area:** Ingestion  
**Status:** Open  
**Depends on:** BE-01

## Why this matters

`docs/OPERATIONS.md` acknowledges the 150-second wall-clock limit on Supabase Edge
Functions and notes that "connectors must be fast or the orchestrator must fan out
across multiple function invocations" — but no fan-out design is specified. Phase 1
targets seven connectors plus HTML scrapers. A full sequential sweep will exceed 150
seconds at any realistic count. Without a design, the first time a developer wires up
the scheduler they will silently time out mid-run with no logged failure.

---

## Prompt

You are building Clyde Culture. Read `docs/OPERATIONS.md`, `docs/INGESTION.md`, and
`docs/decisions/0002-ingestion-runtime.md` (created in BE-01) before proceeding.

The current orchestration model runs all connectors sequentially in a single invocation.
This will exceed the Edge Function time limit as connector count grows.

**Your task** is to design and document a per-connector scheduling model that eliminates
the time-limit problem. If BE-01 resolved to Option B (external Node worker), adapt the
design for that host.

**If BE-01 chose Option A (Edge Functions):**

1. Create `docs/decisions/0003-orchestration-scheduling.md` documenting the decision to
   use one pg_cron entry per connector (one Edge Function invocation per source per day)
   rather than a single omnibus sweep.

2. Create `supabase/migrations/20260601000004_per_connector_cron.sql` with a commented
   example of how to register a connector-specific cron entry:

   ```sql
   -- Example: Ticketmaster runs daily at 02:00 UTC
   -- The Edge Function reads the source slug from query parameters and runs only that connector.
   -- Repeat for each enabled source.
   select cron.schedule(
     'ingest-ticketmaster',
     '0 2 * * *',
     $$ select net.http_post(
       url    := current_setting('app.edge_function_url') || '/ingest',
       body   := '{"source": "ticketmaster"}'::jsonb
     ) $$
   );
   ```

   Add one entry per Phase 1 source (ticketmaster, skiddle, eventbrite, swg3,
   st-lukes, mono, flying-duck). Stagger start times to avoid concurrent DB writes.

3. Update `docs/OPERATIONS.md` to replace the three-option section with the chosen model
   and add a "Adding a new connector schedule" procedure.

**If BE-01 chose Option B (external Node worker):**

Design an equivalent per-connector approach using the chosen external scheduler
(one job per source, or a parametrised job that accepts a source slug). Document it
in `docs/decisions/0003-orchestration-scheduling.md` and update `docs/OPERATIONS.md`.
