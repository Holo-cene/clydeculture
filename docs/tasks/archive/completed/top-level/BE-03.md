## Why this matters

`ARCHITECTURE.md` says event type "is mapped from the upstream category (or guessed via
keyword matching)" but the mapping table (`source_type_category_map`) is empty, the
keyword fallback is never defined, and the step from `tags_guess` to the `event_tags`
junction table is absent from every doc. Without a specified contract, any agent
building the normalisation pipeline will invent these mappings independently and
inconsistently across sessions.

**Step 1 — ✅ DONE — `docs/NORMALISATION.md` already exists:**

This document is the single specification for what the `packages/core` normalisation
pipeline does. It must cover:

**1. Event type classification.** Two paths:
- Primary: look up `(source_id, source_category)` in `source_type_category_map`; use the
  mapped `event_type_id`. Document how connector authors should populate the source category
  field — it should be the raw category string exactly as the API or feed returns it.
- Fallback: keyword matching on the normalised title. Define the keyword list explicitly.
  Suggested starting rules (edit as appropriate for Glasgow context):
  - Title contains `dj`, `club`, `rave`, `techno`, `house`, `disco` → `club_night`
  - Title contains `comedy`, `stand-up`, `standup`, `improv` → `comedy`
  - Title contains `film`, `cinema`, `screening` → `film`
  - Title contains `exhibition`, `gallery`, `art show` → `arts_exhibition`
  - Title contains `workshop`, `class`, `course` → `workshop`
  - Title contains `talk`, `lecture`, `panel`, `discussion` → `talk_lecture`
  - No match → `other`

**2. Confidence score construction.** Define the formula and the JSON structure of
`confidence_inputs`. Include: `source_tier` (int), `has_title` (bool), `has_start_at`
(bool), `has_venue` (bool), `venue_resolved` (bool), `cross_source_corroboration` (bool),
`category_confidence` (string: `mapped` | `keyword` | `fallback`). Define the score:

> **⚠️ SUPERSEDED — historical task context only.**
> The formula below was the original design prompt. It has since been superseded by
> `docs/NORMALISATION.md` Step 4, which is the canonical specification.
> **Do not implement from this formula.** C2 tests (`C2-confidence-red-tests.md`)
> must use `docs/NORMALISATION.md` Step 4 exclusively. Where this formula contradicts
> NORMALISATION.md (base scores, additive inputs, penalty structure), NORMALISATION.md wins.

```
base = source_tier_score(tier)   // tier 1 → 60, tier 2 → 50, tier 3 → 40, tier 4 → 30
field_bonus = sum of: has_start_at +10, has_venue +10, venue_resolved +10
corroboration_bonus = cross_source_corroboration ? +10 : 0
category_penalty = category_confidence == 'fallback' ? -10 : 0
confidence = clamp(base + field_bonus + corroboration_bonus + category_penalty, 0, 100)
```

**3. Tag resolution.** Specify the three steps:
1. Look up each string in `tagsGuess` against `tags.slug` (exact match).
2. If no match, look up against `tags.label` (case-insensitive).
3. If still no match, skip the tag (do not create new tag rows automatically). Log it in
   `confidence_inputs` for review.
Write resolved tag IDs to `event_tags`; update `events.tags_display` as a
comma-separated `tags.slug` string.

**4. Venue resolution.** Call `resolve_venue(venue_name)` first; if null, call
`auto_create_venue()`. Set `needs_review = true` on the event whenever `auto_create_venue`
is called.

**5. Denormalised field population.** List which normalisation step populates each of the
13 denormalised fields on `events` (e.g. `event_type_label` comes from `event_types.label`
where `id = event_type_id`).
