# BE-08: Add a dead-letter path for unparseable items

**Priority:** P2  
**Area:** Observability  
**Status:** Open  
**Depends on:** BE-02

## Why this matters

`IngestResult.errors` is `string[]`. The orchestrator writes these to
`ingest_runs.error_message` — a single `text` column. If a connector returns 80 items
and 5 fail to parse, the 5 failures are collapsed into one string (or silently
truncated), and the raw records that failed are not stored anywhere. There is no way
to inspect the failed items, reproduce the error, or retry without re-running the full
connector. A connector that consistently fails 10% of records will silently
under-ingest with no actionable debug surface.

---

## Prompt

You are building Clyde Culture. Read `docs/INGESTION.md`, `docs/DATA_MODEL.md`, and
`docs/reference/SCHEMA_v5.sql` before proceeding.

**Your task** is to add a structured dead-letter table for per-item parse failures and
update the orchestrator to use it.

**Step 1 — Add `ingest_errors` table:**

Create migration `supabase/migrations/20260601000006_ingest_errors.sql`:

```sql
-- One row per record that failed to parse during a connector run.
-- Enables inspection and retry of failed items without re-running the full connector.
create table ingest_errors (
  id          uuid        primary key default gen_random_uuid(),
  run_id      uuid        not null references ingest_runs(id) on delete cascade,
  source_id   uuid        not null references sources(id),
  error       text        not null,
  raw         jsonb,       -- the raw record that failed, if available
  record_hint text,        -- partial identifier for the record (URL, title, index)
  created_at  timestamptz not null default now()
);

create index idx_ingest_errors_run on ingest_errors (run_id);
create index idx_ingest_errors_source on ingest_errors (source_id, created_at desc);

alter table ingest_errors enable row level security;
-- No public policy; accessed via service role only.
```

Enable RLS (no public policy, service role only — consistent with other ingestion tables).

**Step 2 — Update `IngestResult` in `docs/CONNECTOR_GUIDE.md`:**

Change the error type in the interface definition from `string[]` to structured:

```ts
export interface IngestError {
  error: string;
  raw?: unknown;      // the raw record that caused the failure, if available
  hint?: string;      // partial identifier: a URL, title fragment, or array index
}

export interface IngestResult {
  fetchedCount: number;
  parsedCount: number;
  items: RawEvent[];
  errors: IngestError[];  // was: string[]
}
```

Update the worked example (RSS connector, section 6) to use the new type:
```ts
errors.push({ error: `Failed to parse item "${item.title}": ${String(err)}`, raw: item });
```

**Step 3 — Update `docs/INGESTION.md`:**

In step 5 of the "Scheduled-job model" section, add: "Also writes one row to
`ingest_errors` for each entry in `IngestResult.errors`, storing the error message,
the raw record payload (if available), and a record hint."

Replace the description of `ingest_runs.error_message` (single text field) with:
"A summary error message for the run; full per-item failures are in `ingest_errors`."

**Step 4 — Update `docs/DATA_MODEL.md`:**

Add `ingest_errors` to the "Ingestion Monitoring" section table.
