# DB-03: Document PgBouncer connection strings and prepared-statement behaviour

**Priority:** P1  
**Area:** Ops  
**Status:** Open  
**Depends on:** BE-01

## Why this matters

Supabase exposes two connection endpoints: a direct Postgres connection on port 5432, and
a PgBouncer transaction-mode pooler on port 6543. Transaction-mode pooling does not
maintain session state between statements, which means prepared statements — used by
default in node-postgres v8+ and most ORMs — fail at runtime with
`ERROR: prepared statement "..." does not exist`. This is a silent failure: the client
connects successfully, then errors on the first parameterised query. Neither
`OPERATIONS.md` nor `INGESTION.md` mentions this. Any developer following the existing
docs to wire up ingestion will hit this in production.

The direct connection (port 5432) bypasses PgBouncer but has a hard concurrent
connection limit (10 on Free, 25 on Pro). Edge Functions are short-lived and low
concurrency; the direct connection is the right choice for them. External Node workers
(Option B from `OPERATIONS.md`) have longer-lived connections and should use the pooler
with prepared statements disabled.

This task is documentation only — no schema migration required.

## Prompt

You are building Clyde Culture. Read `docs/OPERATIONS.md` (Environments and Scheduled

**Your task** is to add a "Database connections" section to `docs/OPERATIONS.md` that
prevents the PgBouncer / prepared-statement failure mode.

**Update `docs/OPERATIONS.md`:**

Add a new section "Database connections" immediately after the "Secrets Management"
section and before "Scheduled Ingestion". The section should contain:

1. A table showing the two connection string types, their port, and when to use each:

   | Connection type | Port | Use for | Caveat |
   |---|---|---|---|
   | Direct connection | 5432 | Edge Functions, migrations, one-off admin queries | Hard limit: 10 concurrent (Free) / 25 (Pro). Never use for long-lived pools. |
   | Transaction pooler (PgBouncer) | 6543 | External Node worker (Option B) with long-lived connections | Prepared statements are not supported — must be disabled in the client. |

2. A subsection "Prepared statements and PgBouncer" that explains:
   - PgBouncer in transaction mode does not persist session state
   - node-postgres enables prepared statements by default; they must be disabled when using the pooler URL
   - Code example showing how to disable them for node-postgres:
     ```ts
     import { Pool } from 'pg';
     const pool = new Pool({
       connectionString: process.env.DATABASE_POOLER_URL, // port 6543
       max: 5,
       // Disable prepared statements: required for PgBouncer transaction mode
       statement_timeout: 30_000,
     });
     // For node-postgres, pass `allowExitOnIdle` and use pg-pool without the
     // `prepare` option; or use the @supabase/supabase-js client which uses
     // the REST API (PostgREST) and is unaffected by PgBouncer.
     ```
   - Note that the `@supabase/supabase-js` client uses PostgREST (HTTP), not a direct
     Postgres connection, and is unaffected by PgBouncer.

3. A subsection "Which connection string to use" with clear rules:
   - `SUPABASE_DB_URL` (direct, port 5432): Edge Functions, `supabase db push`, seed scripts
   - `SUPABASE_POOLER_URL` (pooler, port 6543): Option B Node worker running continuously
   - `SUPABASE_URL` (PostgREST API): All `@supabase/supabase-js` calls from connectors and application code

**Update `docs/OPERATIONS.md` Secrets Management section:**

Add `SUPABASE_DB_URL` and `SUPABASE_POOLER_URL` to the list of Supabase project
credentials alongside the existing `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`.
