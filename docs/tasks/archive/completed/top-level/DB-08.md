# DB-08: Add timezone_guess to external_events and IANA validation on events.timezone

**Priority:** P2  
**Area:** Schema  
**Status:** Open  
**Depends on:** —

## Why this matters

Two related timezone gaps need addressing together:

1. **`external_events` has no `timezone_guess` column.** The `_guess` field pattern
   is used for every other extractable field (`event_type_guess`, `availability_guess`,
   `price_min_guess`, etc.) but not for timezone. Tier 1 APIs return explicit timezone
   information — Ticketmaster's `dates.timezone` field, for example. Currently this
   value is discarded; the normaliser infers `Europe/London` from the venue's city. The
   source timezone is preserved in `raw` JSONB but not in the indexed extracted fields,
   inconsistent with the rest of the staging layer design.

2. **`events.timezone` has no IANA validation.** A connector bug or manual INSERT with
   `timezone = 'BST'`, `timezone = 'GMT+1'`, or a typo like `timezone = 'Europe/Lonon'`
   silently stores an invalid value. When the frontend or sync job calls
   `start_at AT TIME ZONE timezone`, Postgres raises `ERROR: time zone "..." not recognized`
   at query time — after the bad data is already in the database.

**Create migration `supabase/migrations/20260601000026_external_events_timezone.sql`:**

**Part 1 — Add timezone_guess to external_events:**

```sql
-- Add timezone_guess alongside the other _guess fields.
-- Populated by connectors when the source provides explicit timezone data.
-- Ticketmaster: dates.timezone (IANA string, e.g. "Europe/London")
-- Skiddle: no explicit timezone field — leave null
-- Eventbrite: timezone field on the event object (IANA string)
-- HTML/iCal scrapers: extract TZID from iCal DTSTART if present
alter table external_events
  add column if not exists timezone_guess text;
```

**Part 2 — Add IANA timezone validation on events.timezone:**

```sql
-- Function to validate an IANA timezone name against pg_timezone_names.
-- Returns the input unchanged if valid, raises an exception if not.
-- Used as a CHECK function via a trigger rather than inline in CHECK constraint
-- (pg_timezone_names is a view, not stable enough for a CHECK constraint in all Postgres versions).
create or replace function validate_iana_timezone(tz text)
returns text as $$
begin
  if not exists (select 1 from pg_timezone_names where name = tz) then
    raise exception
      'Invalid IANA timezone: "%" — use a name from pg_timezone_names, e.g. "Europe/London"', tz;
  end if;
  return tz;
end;
$$ language plpgsql stable;

-- Trigger to validate timezone on INSERT and UPDATE.
create or replace function trg_validate_event_timezone()
returns trigger as $$
begin
  perform validate_iana_timezone(new.timezone);
  return new;
end;
$$ language plpgsql;

create trigger trg_events_timezone_validation
  before insert or update of timezone on events
  for each row execute function trg_validate_event_timezone();
```

> Note: a CHECK constraint referencing `pg_timezone_names` is not reliably STABLE in
> all Postgres versions, so a trigger is more portable. The trigger only fires on
> INSERT or UPDATE of the `timezone` column, adding negligible overhead.
