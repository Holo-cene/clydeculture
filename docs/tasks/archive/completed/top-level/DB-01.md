# DB-01: Add missing schema CHECK constraints

**Priority:** P1  
**Area:** Schema  
**Status:** Open  
**Depends on:** —

## Why this matters

Three invariants that are enforced only in prose or application code need to be
enforced at the database level before ingestion code is written. The most critical is
the published-event invariant: there is no CHECK constraint preventing an event with
`confidence < 60` or `needs_review = true` from being set to `visibility = 'published'`.
Because the public RLS policy exposes all `visibility = 'published'` rows, a bug in the
normalisation pipeline or a mis-typed manual update silently pushes low-quality events
to the frontend. The other two are lower-risk but equally straightforward: `price_max <
price_min` can slip through undetected, and a festival with `start_date > end_date`
silently breaks festival detection.

## Prompt

You are building Clyde Culture. Read `docs/reference/SCHEMA_v5.sql`,

**Your task** is to add three CHECK constraints that enforce invariants currently
described only in prose.

**Create migration `supabase/migrations/20260601000020_schema_constraints.sql`:**

**Constraint 1 — Published-event invariant on `events`:**

An event may only be set to `visibility = 'published'` when `confidence >= 60` AND
`needs_review = false`. Operators override by setting confidence and clearing the flag
before publishing; they do not bypass this constraint.

```sql
alter table events
  add constraint chk_events_publish_invariant
  check (
    visibility != 'published'
    or (confidence >= 60 and needs_review = false)
  );
```

> Note: the threshold (60) is currently hard-coded. This will be externalised in
> BE-19. For now, 60 is the agreed default from `DATA_MODEL.md`.

**Constraint 2 — Price ordering on `events`:**

```sql
alter table events
  add constraint chk_events_price_order
  check (price_max is null or price_min is null or price_max >= price_min);

alter table events
  add constraint chk_events_is_free_price
  check (not is_free or price_min is null or price_min = 0);
```

**Constraint 3 — Festival date ordering on `festivals`:**

```sql
alter table festivals
  add constraint chk_festivals_date_order
  check (start_date is null or end_date is null or start_date <= end_date);
```

**Update `docs/DATA_MODEL.md`:**

> true` raises a constraint violation. Operators override by correcting the event's
> confidence and review state before publishing.
