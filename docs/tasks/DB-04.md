# DB-04: Wire pg_cron extension and scheduled ingestion trigger into migrations

**Priority:** P1  
**Area:** Schema, Ops  
**Status:** Open  
**Depends on:** BE-01

## Why this matters

`OPERATIONS.md` recommends "Option A — Supabase Scheduled Edge Functions" as the Phase
1 ingestion schedule, including "a cron entry in `supabase/migrations/`". But
`SCHEMA_v5.sql` only enables `pgcrypto` and `pg_trgm`. There is no
`CREATE EXTENSION pg_cron` and no `cron.schedule()` call anywhere in the migrations.
`pg_cron` is available on all Supabase plans but must be explicitly enabled. Without
this, Option A cannot be implemented from the migration files, and a developer following
the docs will discover the gap at deployment time.

This task depends on BE-01 (runtime decision) because the cron trigger target changes
depending on whether Option A (Edge Function) or Option B (external worker) is chosen.
If BE-01 resolves to Option B, substitute the pg_cron trigger with documentation of the
external scheduling mechanism. If it resolves to Option A, implement the trigger as
specified below.

### Option A path — Edge Function triggered by pg_cron

**Create migration `supabase/migrations/20260601000022_pg_cron_ingestion_schedule.sql`:**

```sql
-- Enable pg_cron. The extension must be enabled in the extensions schema.
-- On Supabase, pg_cron is pre-installed but not enabled by default.
create extension if not exists pg_cron with schema extensions;

-- Grant usage so the cron scheduler can invoke functions.
grant usage on schema cron to postgres;

-- Schedule the ingestion orchestrator Edge Function once daily at 06:00 UTC.
-- The Edge Function URL follows Supabase's standard pattern:
--   https://<project-ref>.supabase.co/functions/v1/ingest
-- Replace <project-ref> with the actual project reference from the environment.
-- The Authorization header uses the service role key, stored in Vault.
--
-- This cron job is created once. To change the schedule, update this migration
-- or use: select cron.alter_job(jobid, schedule => '0 8 * * *') after deployment.
select cron.schedule(
  'daily-ingestion',        -- job name
  '0 6 * * *',              -- 06:00 UTC daily
  $$
  select
    net.http_post(
      url     := 'https://<project-ref>.supabase.co/functions/v1/ingest',
      headers := jsonb_build_object(
        'Content-Type',  'application/json',
        'Authorization', 'Bearer ' || current_setting('app.service_role_key')
      ),
      body    := '{}'::jsonb
    );
  $$
);
```

Also enable the `pg_net` extension which is required for HTTP calls from pg_cron:

```sql
create extension if not exists pg_net with schema extensions;
```

Add a comment block at the top of the migration:

```sql
-- IMPORTANT: Before applying to production, replace <project-ref> with the actual
-- Supabase project reference. Store the service_role_key in Vault and reference it
-- via current_setting('app.service_role_key') — never hard-code it here.
-- To verify the job was created: select * from cron.job;
-- To check recent runs: select * from cron.job_run_details order by start_time desc limit 20;
```

**Update `docs/OPERATIONS.md`:**

In the "Scheduled Ingestion" section, replace the three-option conditional with a
single statement matching the chosen runtime (referencing ADR 0002), and add:

> The cron job is defined in `supabase/migrations/20260601000022_pg_cron_ingestion_schedule.sql`.
> To inspect schedule state in production:
> ```sql
> select jobname, schedule, active from cron.job;
> select * from cron.job_run_details order by start_time desc limit 10;
> ```

### Option B path — external cron (documentation only)

If the ADR chose Option B, update `docs/OPERATIONS.md` Scheduled Ingestion section to:

1. Remove the conditional three-option language
2. State that ingestion is triggered by [chosen host: GitHub Actions / Render / Railway]

3. Provide the cron expression (`0 6 * * *` or equivalent) and where it is configured
4. Note that pg_cron and pg_net are not required and are not enabled in migrations

## Acceptance criteria

**Option A path:**
- [ ] Migration enables `pg_cron` and `pg_net` extensions in schema `extensions`
- [ ] `cron.schedule()` call schedules the Edge Function at `0 6 * * *`
- [ ] Migration has a comment block flagging the `<project-ref>` placeholder
- [ ] `docs/OPERATIONS.md` no longer has conditional three-option language
- [ ] `docs/OPERATIONS.md` shows how to verify the cron job and inspect run history

**Option B path:**
- [ ] `docs/OPERATIONS.md` states the chosen external host and trigger mechanism
- [ ] No pg_cron migration file is created
- [ ] `docs/reference/SCHEMA_v5.sql` is not modified
